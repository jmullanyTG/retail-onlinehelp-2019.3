Creating a Rate of Sale (ROS) scenario

The first component is where users create a Rate of Sale scenario which defines the inputs required to model historic data in which to generate rates of sale outputs.  Users can define and save rate of sale scenarios for application now or in the future based on existing placeholder/variant and site filters – or define filters dynamically to suit your needs and specified time frames. Users can also apply Min/Max Price and Max Price Discount filters to limit the data being considered during calculation.

Additionally, users can easily recall saved scenarios by filtering the list with placeholder/variant and site filters.  Time frame selection can be static or dynamic, with the ability to name dynamic ranges (i.e. Last 4 weeks) and dynamic scenarios will recalculate regularly to make use of the most current historical data.  Further filtering against the historical data is available to refine the ROS scenarios by defining minimum price, maximum price, maximum discounts and price types as restrictions on what is and is not included in the data set.

To create a ROS scenario

  1. Variant and Site Filter selection
    1. Select an existing placeholder/variant filter or create a new filter to select the items for the scenario (this determines which scenarios will appear after scenarios have been created and saved)
    2. Select an existing site filter or create a new filter to select the sites for the scenario (for application in assortment planning, select (All Sites)

  2. Click the Execute button

  3. Click the Add button to create a new scenario line.

  4. Complete the fields as required.

  5. Click Save to save the scenario inputs.
  6. Click Calculate to calculate the scenario lines.

Field reference

Field Description
Name A name for the scenario (a default name will populate but can be overridden)
Description A description for the scenario
Scenario Mode Select Assortment Planning
Assortment Plan Select the active assortment plan
Scenario Type Select Revenue or Sales Units to model against
Product Filter Select the product filter for the scenario, this will drive which items are included in the calculation
Site Filter The sites for the scenario
Cluster Set Select the cluster sets to be included for the scenario, this will drive which clusters are included in the calculation. Cluster sets must be configured for use with Assortment Planning. (For more information, see Cluster Set Configuration Screen.)
Time Frame Select an existing time frame or create a new time frame for the scenario. (For more information, see Selecting a Time Frame.)
Sales Types Select the types of sales to include in scenario (Reg, Markdown, Promo, etc)
Include Values Outside Phase In/Out Raw ROS includes Outer Zeros by default.

If phase in/out is set to the system default, this calculation uses the timeline specified in the scenario instead. Phase in will be first sale within the time frame and phase out date will be the last sale within the time frame.

For example, Timeframe is past 365 days and last sale was past 20 days. The 20 days from last sale to today are considered outer values.

Sometimes, final week of partial-period will be deliberately dropped in case ROS calculation was run mid-week and half a week of sales can cause a false drop in sales. For historical data, it could easily indicate a stock out as well. We were selling product, then ran out, so sales stopped. Thus, we ignore the bucket that phase out / last sale occurs inside of when calculating ROS.

  • Exclude: excludes all sales history outside of phase in/out days
  • Include: include all sales history outside of phase in/out days
  • Include If Non-Zero: If there are non-zero sales outside of phase in/out days, include those
  • Include If Non-Zero Or Site Has Stock On Hand: Condition of Include If Non-Zero in addition to include 0s if there are SOH
Include Inner Zeros

Raw ROS includes Inner Zeros by default. If phase in/out is set to the system default, this calculation uses the timeline specified in the scenario instead. Phase in is the first sale within the time frame and phase out date is the last sale within the timeframe.

  • Exclude: exclude weeks in the timeframe with 0 sales.
  • Include: includes weeks in the timeframe with 0 sales.
  • Exclude If No Stock On Hand: If SOH was 0 for that week, it excludes the week. Otherwise, it includes the week as 0 sales.
Min Weekly Sales Deletes individual inventory history weeks if they do not meet this threshold, rather than whole products based on least single week. The input is inclusive. For example, if the input is 1, then ROS considers weeks with 1 unit sold.
Days From Phase In

This limits Adjusted ROS to a certain number of days since product phase-in. This also rounds up all days into weeks. So, 8 days from phase in would be considered 2 weeks.

If the inventory phase in date is set to system default, days will start counting from the earliest date specified in the Time Frame column.
Min Price

Input the lowest price of variants to include in the scenario; set to zero to ignore this parameter. During calculation, the Min Price specified is compared against the selling price by period, as defined in sales history by type, for all sales included in the Product Filter, Cluster Set, Sales Types, and Time Frame parameters of the scenario; any sales made at a selling price below Min Price will be excluded.

Selling prices by period do not necessarily correlate to average selling prices shown in the UI (for example, AUR within Assortment Planning).
Max Price Discount

Input the max % discount to ticket price of variants to include in the scenario; set to zero to ignore this parameter

During calculation, the Max Price Discount specified is compared against the selling price by period, as defined in sales history by type, for all sales included in the Product Filter, Cluster Set, Sales Types, and Time Frame parameters of the scenario; any sales made at a selling price below (Max Price Discount * ticket price (full undiscounted price) of the variant in question) will be excluded.

Selling prices by period do not necessarily correlate to average selling prices shown in the UI (for example, AUR within Assortment Planning).
Price TypeDMDEV-56045

Select the price type to include in the scenario.

Price type is only valid with Max Price, Min Price, and/or Max Price Discount are specified; price types are defined by your organization during software implementation and can be things like MSRP, Clearance, etc.

The difference between Sales Type and Price Type can be difficult to understand, but basically, think of Sales Types as defining the larger bucket of sales from which to draw. Price Type is an optional data point that can change by period and can be used to further limit the data selection for the scenario.

For example, it is possible to include the Sales Type Markdown but then further limit that data by setting a Price Type of Clearance and a Max Price Discount of 75% to only capture variants that had the Clearance Price Type and selling prices that were no more than 75% the original ticket price for each period in the scenario time frame.
Access Level Select access level to the scenario: Public, Private, or Shared. If shared, select the shared users by clicking on the ellipsis.
