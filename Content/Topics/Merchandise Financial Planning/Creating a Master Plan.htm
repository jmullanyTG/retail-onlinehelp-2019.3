Creating a Master Plan

When working in the financial plan maintenance view, the first action our clients need to take is to create a master plan.  A master plan is a Financial Plan that is created in the MFP module and is not editable.  This plan is considered the “single” version of the truth.  Master plans should be used as approved plans within the planning process.  Master plans contain historic and plan periods.  Historic periods represent “what actually happened” within the business and plan periods are the approved plans for future periods.  In managing a Master plan it is recommended that a super user or system administrator has control over this feature within the system.  To create a master plan see the below example.

To create a master plan within the financial plan maintenance view, click on the “New” icon located in the top left portion of this view.  Once the user has clicked on the “New” icon, the maintain financial plan window will appear on screen.

The first tab a user works with in this view is the General tab.  The general tab is where the user will give the master plan a name and description, define the plan type, the currency to work with, how the plan will be seeded, the calendar to plan with, the level of the calendar, and finally the start period.  These general setting form the basis of the structure of the master plan. 

Name:  This is free form text and describes the name of the plan

Description:  This is free form text and provides the opportunity to describe what the plan is and any other important information about the plan that may be important to the user group.

Plan Type: The plan type defines the metric set and calculations that will make up the master plan.  There are three main plan types to choose from.

Location Plan: This plan provides the ability to plan by location at defined levels of the product hierarchy.  This plan contains the ability to plan sales, margin and Open to Ship to support the location planning process.

Hybrid Cost-Based Plan: This plan provides the ability to plan by Channel/location groups at defined levels of the product hierarchy.  This plan contains the ability to plan sales, cost of sales, margin, inventory, receipts, and open to buy to support the budgeting process.  This plan is for clients who value their inventory at cost.

Hybrid Retail-Based Plan: This plan provides the ability to plan by Channel/location groups at defined levels of the product hierarchy.  This plan contains the ability to plan sales, cost of sales, margin, inventory, receipts and open to buy to support the budgeting process.  This plan is for clients who value their inventory at retail.

Hybrid Cost Dual-Markdown Plan: This plan type is for organizations who wish to plan their permanent and promotional markdowns separately. Promotional markdowns are tied directly to Promotional Sales and reduce AUR for periods in which they are realized, impacting margin through that reduction. Conversely, because markdown sales do not necessarily occur in the same period in which the actual markdowns are taken, Permanent Markdowns are managed strictly as a percentage of sales, and are subtracted directly from the Maintained Margin value for the period in which they occur. In this plan type, average unit cost (AUC) and average unit ticket (AUT) do not typically change from period to period because of changes to the sales plan. AUC will change as a result of incoming receipts or other changes to the inventory mix. AUT will only change if the user changes it directly or indirectly, through a change in IMU %.

Plan Currency:  This selection defines the base currency in which the users will be working with, all currencies are supported in this version.

Management Type: determines whether a plan will be created within Demand Management (Manual type) or data for the plan will be imported via API (Import type); set to Manual by default

if an import post runs for a plan of type Manual, the plan will not be affected and an error will be raised

when import posts run for plans of type Import, hierarchy levels will automatically be added to the plan based on the import data; if desired, the API can be run in “Window” mode, deleting all existing data in a plan and re-loading it from the import data

Seed Type:  This selection defines how future periods in the plan will be seeded for the metrics offered within the plan type that has been selected.  There are three standard seeding options to choose from.  Seed Types in our standard offering cannot be combined to seed future periods when creating the plan.

Historic Sales LY:Selecting this option will seed future periods with sales, sales units, and average unit retail from last year’s comparable periods.

Historic Sales LLY: Selecting this option will seed future periods with sales, sales units, and average unit retail from two years ago comparable periods.

Forecasted Sales: Selecting this option will seed future periods with sales units that are coming from the forecast module.  Sales are derived by multiplying the forecasted sales units times the average unit retail from last year’s comparable periods.

Important :regardless of the selected Seed Type, metrics that would be seeded with negative values based on the available data indicated by that type will be seeded with zeroes instead

Calendar:   This selection defines the calendar that users will be working with in the master plan.  We offer two standard selections based on a retail planning calendar.  Users can select a 4-4-5 calendar or 4-5-4 calendar.  These calendars are NRF retail planning calendars.  It is possible to import custom calendars during implementation of this module if necessary.

Calendar Level:  This selection defines the lowest level of time that users will be working with in the master plan.  There are three standard options a user can select.  (Year, Month, Week)  Most plans created in this module go to the Month or Week level.

Start Period:  This selection defines the first period contained within the plan.  This can be a historical period or future period.

Stock Split Supply Chain: The supply chain used to determine how inventory is split among DCs. (The Supply Chain Status field indicates whether the supply chain is valid for use with this plan.)

Stock Split Sales Types: The sales types to consider when apportioning stock to DCs.

Supply Chain Status: This read-only field indicates whether or not the supply chain is valid. (See Validating a Supply Chain for more information.)

Owner:   This will be greyed out until the plan is created.  Once the plan is created the system will display the name of the user who created the plan.

Creation Date:  This will be greyed out until the plan is created.  Once the plan is created the system will display the date the plan was created.

Status:   This will be greyed out until the plan is created.  Once the plan is created the system will display the status the plan is in.  Plan status will be covered in the Plan approval section of this document.

The second tab a user works with in this view is the Product tab.  The Product tab is where the user will define the product hierarchical structure contained within the plan.  The product hierarchy levels selected define the levels in which users can work with.

The top window of the screen is where the plan levels are selected.  In the example above, the lowest level of this master plan will be category.

The bottom window of the screen is where the user will determine which categories will be contained within the master plan.

 

The third tab a user works with in this view is the Channel tab.  The Channel tab is where the user will define the location hierarchical structure contained within the plan.  The location hierarchy levels selected define the levels in which users can work with.

The top window of the screen is where the plan levels are selected.  In the example above, the lowest level of this master plan will be channel.

The bottom window of the screen is where the user will determine which channels will be contained within the master plan.

The fourth tab a user works with in this view is the User Access tab.  The User Access tab is where the user will define the user/role access permissions within the plan.  Access permissions can be defined by role within the organization or by User.  In the bottom window of this view, setting up the permissions requires that the user navigate through three tabs.  (Product, Channel, Merge Measure Set).  In the example above the highlighted user has read and write access to Women’s Denim.  This means that the user can view and edit Women’s Denim for any branch created that is connected to this master plan.  For the master plan, the user will have read and merge rights for Women’s Denim. 

Access Permissions are also defined by channel.  The user can click on the channel tab and define the level of the location hierarchy roles and users have for read and write access.  In the example above the highlighted user has read and write access to US Retail and E-Com channels.  This means that the user can view and edit Women’s Denim for the US Retail and E-Com for any branch that is connected to this master plan.  For the master plan, the user will have read and merge rights for Women’s Denim.

Access Permissions control which measure sets can be merged into the master plan from a branch that belongs to that master plan.  Measure sets are defined in the Header Area controls section.  In the example above the highlighted user has permission to merge all measures within the Hybrid Cost-Based Plan type into the master from the branch or branches they are working with.

The last tab a user works with in this view is the View Collection tab. The View Collection tab defines the collection of metrics that are contained within the plan as well as the collection of metrics that are contained within any snapshot taken of the plan.  View collections are set up by clicking add for both the top and bottom window in the view.  Upon clicking add, the user will select the Plan Type and View Collection set that is to be associated with this plan.  Once this screen is completed, the user will click OK and that will trigger the system to create the plan based on all the criteria that was defined in the five tabs contained in the maintain financial plan window.  Upon clicking OK, the system will return the user back to the Financial Plan Maintenance view and will display the plan that has been created.

  • plan view collection sets defined in this tab are inherited by default by any Branch plans created from a Master plan. If the same view collection sets assigned to a Master plan are desired for use in Branch plans beneath it, this section can be left blank. Only if different view collection sets are desired must the user make changes to this section.

The Master Financial Plan that has been created will update to “Created” state.  This means the plan is ready to be initialized.  Initializing the master plan basically populates all the measures defined for historical and future periods.  A plan must be initialized to view and work with the Master Plan.  To initialize a plan the user will simply click the Initialize icon in the header of the Financial Plan Maintenance view.  Once the user clicks initialize a process will run and return the user back to the Financial Plan Maintenance view and the Master Financial Plan will be assigned the status of Initialized.
