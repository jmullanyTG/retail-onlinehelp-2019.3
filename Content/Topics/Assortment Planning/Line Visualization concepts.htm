Line Visualization concepts

Line Visualization allows you to view and refine your plans at multiple levels whether or not you know the exact items to be ordered. Here, you can assort items to site clusters, craft your sales plans via rate of sale, total sales, or total targeted buy quantity in both revenue and units, plan receipt flow strategy, and even manage your sales and inventory plans pre- or in-season by period.

The following are some key concepts to be aware of when working within Line Visualization:

  • Maximum Data Rows – the system policy Line Visualization Maximum number SQL Data Rows, accessible via Maintenance > Policies > Module: Assortment, determines the maximum number of SQL data rows that can be queried in Line Visualization

An error displays if the data selection parameters exceed this maximum.

  • Placeholders and Variants – whether you seed your plan via the Smart Start process or choose to manually add lines to you plan and build from the ground up, placeholder items are your starting point. Think of placeholders as slots into which actual styles can be placed. We refer to actual styles as variants. All aspects of your assortment plan can be managed regardless of whether variants are attached to placeholders, and when variants are attached, all your work remains intact
  • Assigning Variants – Assortment Planning can accept an external data feed from a PLM (Product Lifecycle Management) or similar system to define variants within Demand Management. When this feed is configured, assigning variants is as simple as dragging and dropping them from the Style Bank (see Style Bank section below). Additionally, if the integrated PLM is able to send product attribute information attached to the variant data, those attributes will be passed on to placeholders to which those variants are assigned.

  • Placeholders and Collections – it is possible to plan at either the placeholder (equivalent to style-color) or collection (equivalent to style) level by using the view mode for each. While the most typical case for planning in collection mode is to plan multiple colors of the same style together, Demand Management Assortment Planning does not restrict collections to this configuration. Collections can be comprised of multiple styles within the same product level or even across product levels.

Collections comprised of styles that cross multiple product levels may not be visible in their entirety to all users, based on their product hierarchy permissions. In this scenario, the user will only be able to view and work with the portion of the collection to which they have appropriate permissions.

  • Clusters versus Sites – In order to maximize system performance, Assortment Planning was designed to operate primarily at the site cluster level rather than the site level. However, it is still possible to make site level product exclusions, saving you the effort of having to create custom clusters every time you have a scenario where a particular variant should not be assorted to specific locations within existing clusters. See Location Exclusions below for more information.

  • Calculation Model – the Line Visualization calculation model is built on single directional locking calculations that change all dependent metrics and balance to all levels of the plan regardless of which level they were changed at. In most cases, users are able to see the effects of their changes immediately without having to Apply them first. However, there are some calculations (such as when AUT (average unit ticket), phase in, or phase out are changed) that require background processing and an Apply to complete.

    Prior to Apply, the system will lock (indicated by a red or pink conditional format to the line, depending on the row color) any metrics for the placeholder or collection that are dependent upon changes already made:

    In this state, the user can continue to change unlocked metrics without Applying. All metrics will become unlocked again after clicking Apply.

  • Tracking Against Ordering Deadlines

    As one of the primary purposes of Assortment Planning is planning future orders so it is essential to be able to track products you plan to order against relevant deadlines. Once commitment orders are generated for products within your plan, the workflow attached to those orders drives key dates in the ordering process, which can be tracked through Order Manager and the Assortment Upcoming Commitments Dashboard, but prior to commitments being created, assortment planning reads default lead times by product hierarchy level and suppliers associated with variants to provide visibility to initial order due dates. The following needs to be set up to benefit from this information:

    • Default Lead Time Days by Product Hierarchy Level: accessible via Defaults within the Manage Assortment Plans dialog, the Lead Time Days field allows users to set generic lead time days by product hierarchy level

    • The system uses this setting to determine the number of days before a receipt date an order must be placed to arrive on time when there is no variant specific supplier lead time available, either because no variant is assigned to a given placeholder or because the assigned variant doesn’t have a supplier and lead time associated. For more information on setting defaults, see Managing Assortment Plans and setting plan defaults.

  • Variant Supplier Assignments: provided supplier information has been imported into Demand Management, variant supplier assignments can be set through Variant Manager (Manage > Variant Manager)

    • The Supplier Code / Lead Time field can be used to select from available suppliers for variants listed in the active Variant Manager Document. The system uses the lead time days associated with the selected supplier to determine the number of days prior to a receipt date when that variant is assigned to a placeholder within Assortment Planning

    As with other fields in Variant Manager, the Supplier Code / Lead Time field can be set for multiple variants at once by using CTRL + Click or Shift + Click to select the desired variant lines then using the right click > Multiple Changes functionality.

  • Actualization – because Assortment Planning is an in-season planning tool and a pre-season tool, as periods within your plan pass, plan values will be replaced by actual values during regularly scheduled system processes. As this occurs, certain plan metrics and functions will be locked for placeholders whose lifecycles intersect with actualized periods:

    • Collection % Contribution, Ttl Buy $, Ttl Buy U, Min Pres, AUT, and AUC will not be directly editable

      • Collection % Contribution, Ttl Buy $ and U can still be impacted by changes to ROS $, ROS U, Ttl Sls $, Ttl Sls U, and S/T %

      • AUT can still be changed via the AUT side view, though changes can only be made to non-actualized periods; in the main planning and cluster set views, AUT will display the weighted average of AUT values for all periods, both actualized and future, beneath

      • AUC can still be indirectly impacted through the Receipt AUC of planned future receipts, as well as the cost of on order units and actual receipt units. In the main planning and cluster set views, AUC will display the weighted average of AUC values for all periods, both actualized and future, beneath

    • The Split function (see Header Area Controls below), which is used to split placeholders into two or more placeholders with shorter lifespans, will be disabled

    • Additionally, for metrics that remain editable, constraints will be put in place to prevent users from making changes that would cause sales to drop below actualized totals

    If a user attempts to make a change that would violate this, they will receive an error and the changed metric(s) will be reverted to their previous values

  • VAT Actualization – as periods begin to actualize in your assortment plan, so VAT amounts are actualized for those periods. If your organization deals with VAT, it is important that your data import includes Sales ex VAT, as the system uses the difference between Sales and Sales ex VAT to calculate historic VAT % for actualized periods. If any records imported for a given variant/cluster/period lack Sales ex VAT, the system will null the VAT % for the entire variant/cluster/period, as an indicator to the user that there is missing data in the import. This has the potential to cause anomalies in AUR ex VAT in the Main Planning View and with metrics in the Price Exceptions View

A way to check for this is to look at the VAT percentages in Sales Plan view for periods that have actualized VAT % of zero though VAT percentages were in effect for the product.
