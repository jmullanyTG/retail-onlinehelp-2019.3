Receipt Plan View

Receipt Plan view allows users to identify the ideal receipt strategy for items they are planning to purchase. Functionality includes the ability to:

  • set constraints, such as minimum Periods of Supply
  • define the periods in which receipts are allowed to arrive and how many receipts are needed
  • send commitment orders to Order Manager for final processing, review, approval, and submission to external systems.

Toolbar functionality

Parameters: Displays or hides the parameters pane at the top of the view.

Parameter Description
View Mode

The view mode for receipt plan view:

  • Total Company: the revenue and unit metrics as total company values – meaning the total volume for all sites contained in whatever level the Pivot Grid is aggregated to
  • Average Store: the revenue and unit metrics as average store values – meaning the per site volume for the number of sites contained in whatever level the Pivot Grid is aggregated to

Execute: Refreshes the view. This is not typically necessary, as the view refreshes on its own when a new placeholder/collection is selected in the main planning view, when changes are applied, and when the View Mode is changed.

Other options

  • Save: Saves all changes
  • Cancel: cancels all changes made since the last Apply and reverts them to their previously saved values
  • Export: Exports and saves the Pivot Grid in many common formats including Microsoft Excel and Adobe PDF
  • Open: Opens the Pivot Grid in many common formats including Microsoft Excel and Adobe PDF and does not require a save prior to viewing
  • Graph: Graphs the selected data in the view

  • Fields: Allows the user to customize the field layout for the view. Custom layouts can be saved and restored by right-clicking in the Pivot Grid and selecting Save Layout or Restore Layout, respectively.

Notes

Editable Metrics

Denoted by light blue row coloration, these are the only metrics that are editable by the user in Receipt Plan view; all others are calculations or reads from other sources. Changing Minimum Presentation in this view WILL update Min Pres in the Main Planning and Cluster Set views, and in Sales Plan View.

Changing Receipt Units in this view WILL NOT update metrics in the Main Planning or Cluster Set views – this is by design, as receipts are suggested in Assortment Planning via a complex algorithm considering sales and all user-specified constraints; because sales are considered in the output already, to then, go update Ttl Buy U would cause the sales plan to change, thus potentially invalidating the results of the receipt calculation. It WILL, however, update Receipt Units in Sales Plan view, as will changing Receipt AUC.

Calculations

Receipt plans are calculated via the Receipt Planning Engine, which uses a complex algorithm to determine the optimal receipt flow based on sales and minimum presentation by period and user-defined constraints; because of this, system receipt recommendations require user action to recalculate, rather than recalculating dynamically as changes are made in the user interface. Recalculation can be done either for the selected placeholder/collection only by clicking Recalculate Receipts in this view, or for all placeholders/collections that require recalculation by clicking Recalculate in the Main Planning view header area.

Receipt Engine Outputs

The System Receipt Units and Ideal EOP Units metrics are outputs of the Receipt Planning Engine, which must be re-run when any of the inputs it considered change. These fields will be conditionally formatted any time a change is made that could invalidate the results of the last engine run – changes to sales, min pres, shrink, inventory adjustments, or receipt constraints all fall into this category.

Plan Horizon vs. Receipt Planning Engine Horizon

If the Plan Horizon, set at the time of plan creation, extends beyond the receipt planning engine horizon for a given item, data that extends past the receipt engine horizon appears in blue italics with zero values. See example in Plan Horizon vs. Replen Forecast Horizon below.

Replen Placeholders/Collections

When a placeholder/collection is of the status Replen, the system pulls all relevant data to form its sales plan from the Forecasts module. Users should be aware of the following differences in this view when dealing with Replen items:

Read Only: All normally editable metrics in Receipt Plan view become read only being populated directly from aggregated replenishment forecast data.

Receipt Constraints: All receipt related constraints are managed within the Forecasts module, thus, receipt constraints in Assortment Planning for Replen placeholders/collections are cleared. With a change of status to something other than Replen, the default constraints assigned to any corresponding product level is reinstated.

Receipt Periods: For Replen placeholders/collections, the necessary orders to support forecasted sales are generated from the Forecasts module and sent to Order Manager for processing. Users do not need to set Receipt Periods in this view the way they do for non-Replen items. Receipt Periods are set automatically based on when the generated orders are scheduled to arrive.

Plan Horizon vs. Replen Forecast Horizon: if the Plan Horizon, set at the time of plan creation, extends beyond the replenishment forecast horizon for a given item, the system conditionally formats the periods that go beyond the replenishment forecast as shown below to draw the user’s attention to this. System Policies

A number of system policies affect the output of the Receipt Planning Engine. These system policies are managed in Maintenance > Policies. However, this should not be done without the assistance of a Demand Management technical consultant. The one possible exception is the following:

Assortment Engine Lead Time: This setting determines the number of days forward from receipt date that received stock is assumed to be available to support sales. The default is 7 to assume receipts are available to sell one week after receipt. This can be adjusted based on a realistic assessment of the capabilities of your supply chain. When the engine is calculating receipts, this setting essentially determines the range of periods it must consider to meet the defined constraints for a given period. See examples for Periods of Supply and Min Stk to Sales below for details.

Allow Receipts After Markdown: (Maintain > Policies > Engine: Commitments) When set to False (default), the Receipt Plan only considers the regular portion of the Sales plan and ignores the markdown phase of the plan. Set the system policy to True if additional stock should be ordered beyond the planned markdown date.

Working with Receipt Plans

  • Recalculate Receipts: Re-runs the Receipt Planning Engine for the selected placeholder/collection and populates System Receipt Units and Ideal EOP Units with the results. Receipt plan default constraints can be set via Manage Assortment Plans > Defaults (see corresponding section above for details).

    System policies affecting the Assortment Engine output can be set via Maintenance > Policies > Engine: Commitments (see Line Visualization Supporting Views – Receipt Plan below for details).

    Receipt recalculations require complex calculations and, thus, are sent to the Workload Queue module for scheduling and processing. Though receipts are recalculated immediately, they may be deferred for later calculation based on the number of placeholders requiring recalculation and other jobs currently in the queue. Placeholders awaiting recalculation are locked and conditionally formatted to indicate this.

Assortment Receipt Plan Constraints grid

  • Periods of Supply: Sets the periods of supply (period type as defined by the current assortment plan) for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs. Actual by period constraint is defined as the sales planned for this many periods forward from a given period. This setting interacts with the system policy Assortment Engine Lead Time.

    Assortment Engine Lead Time is set to 7 days, Periods of Supply is set to 2, and I am planning a receipt to arrive in Wk 1. For this constraint to be considered fulfilled for Wk 1, I must end the period with at least 60 units of inventory on hand to cover weeks 2 and 3. The lead time of 7 days stipulates that stock being received in given week cannot be used to cover the sales planned in that week, thus Periods of Supply begins calculating need at Wk 2 rather than Wk 1.

  • Periods Before Phase In: Sets the default number of periods prior to Phase In date that initial receipt quantities for the placeholder/collection instance should be scheduled to be received

    I have a placeholder instance with a Phase In date of wk 4 and a Periods Before Phase In of 2 - my default calculated initial receipt for the placeholder instance will be in wk 2

  • Flow Cadence: Sets the default number of periods between receipt dates calculated by the receipt planning engine

    I have a placeholder instance with a Phase In date of wk 3, a Periods Before Phase In of 2, and a Flow Cadence of 4 - if I have 3 receipts, they will occur in wk 1, wk 7, and wk 11. This is because the purpose of Periods Before Phase In is to ensure product can be in stores by the Phase In date, thus the first flow date is considered to be the Phase In date for the purposes of receipt engine calculation

  • Min Stk/Sales (Minimum Stock to Sales): Sets the minimum stock to sales ratio for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs. Actual by period constraint is defined as the sales planned for a given period multiplied by the constraint value (period sales of 10 * Min Stk/Sls of 4 = EOP stock requirement of 40).

  • Shrink % of Sales: user editable positive percentage to 2 decimal places (1.52%). Sets the anticipated shrink units per period as a percentage of planned period sales units for the placeholder/collection instance as a minimum stock constraint for receipt engine outputs

    If I have a period with planned sales of 500 units and a Shrink % of Sales of 2.00%, I must have an EOP units of at least 10 for this constraint to be satisfied for the period

  • Target S/T %:user editable, has no calculation effect or relationship, defaults to S/T %.
  • S/T %: read-only. percentage (10.1%); sets the target sell-through percentage for the placeholder/collection. Changes to this metric alter Ttl Buy U to achieve the entered S/T %. Calculation: (Ttl Sls U / Ttl Orders U).
  • Recalc Receipts: Sets whether System Receipt Units and Ideal EOP Units will be recalculated for the placeholder/collection instance when Recalculate Receipts in Receipt Plan view or Recalculate in the Main Planning view is clicked
  • Hold Receipts: Whether the current total Receipt Units for the placeholder/collection instance can change to accommodate Placeholder Cluster Constraints upon recalculation.

    If unchecked, the system calculates the receipt strategy to meet the specified constraints for as long as possible without increasing Receipt Units by moving receipt units forward in time

    If unchecked, there will be scenarios where the system will not be able to satisfy user defined constraints for the life of the item within the cluster

  • Site Count: non-editable integer; count of all sites the placeholder/collection is assorted to at whatever level the Pivot Grid is aggregated to for the period

    if multiple cluster set snapshots are effective during the life of the placeholder/collection, this number can vary by period

  • Sales Units: non-editable integer; planned/actual sales units for the period; read from Sales Units (user, not System) in Sales Plan view

    Calculation: (Sales / AUR)

    Calculation: From Receipt Plan.   • Commit: Sets the periods for which to create commitment orders when Create Commitments is clicked. This field is checked by default for checked Receipt Periods and cannot be checked for periods where Receipt Period is not checked
  • Order By Date: non-editable date (10/2/2017); displays the order deadline for the corresponding Receipt Period/Units based on assigned default lead times; essentially, the first day of the receipt period minus Default Lead Days from the Main Planning view

  • Receipt Units: Planned/actual receipt units for the period. Defaults to System Receipt Units unless user has manually entered a quantity and Applied

    Calculation: System Receipt Units – Committed Units – On Order Units – On Order (Unknown) Units

    Once a commitment order has been created (see Create Commitments, above), the quantity that was in this field will move into the Committed Units field, indicating that an order has been generated for the planned quantity; as the order moves through the commitment process, the units will transition or On Order Units (or On Order (Unknown) Units) and then ultimately show again as receipt units when the period in question actualizes, provided the units were actually received in that period; this ensures that units for the same order are not double-counted as they move through the process.

    Once the above described transition occurs, it is still possible to add more Receipt Units to this field and place subsequent commitment orders for them as needed; this may happen automatically as your sales plan changes as a result of receipt plan recalculations but you can also simply enter the desired additional quantity in the field then again clicking Create Commitments when ready; each individual order will contribute to the totals shown and be tracked and transitioned through the applicable fields individually

    Helpful Hint: Entering Receipt Units at the Grand Total level in this view is a quick way to balance your receipt plan to the Ttl Buy U in the Main Planning and/or Cluster Set views if desired; simply aggregate the pivot grid to the desired level by removing fields as needed (ie. Cluster Set, Cluster, Style, etc.) and enter the Ttl Buy U value in the Grand Total column at the far right – the structure of your receipt plan, including flow dates and % to total of receipt units by period, will be respected while the system balances the period level entries to match the new Grand Total

  • Receipt AUC: user editable currency ($1.01); sets the average cost per unit for receipts in the period

  • Receipt Cost: non-editable currency ($100); planned/actual cost of receipts for the period

    Calculation: (Receipt Units * Receipt AUC)

  • Committed Units: non-editable integer; units for which commitment orders have been created but which do not have corresponding confirmed purchase orders. Auto-populated when commitments are created for periods having Receipt Units; read from commitment orders for the corresponding variant/cluster/period in Order Manager. Receipt units for a given period are transitioned out of Receipt Units to Committed Units when commitments are created for that period.

    Commitment Units can be automatically decremented based on the presence of confirmed On Order Units but only if the system policy Assortment Order Type is set to Variant Orders and Demand Management has integrated with an ERP/POM system capable of returning order data with the UUIDs Demand Management generates for each order sent. If this is not the case, a custom procedure is needed to appropriately decrement Commitment Units and avoid duplication of units with On Order (Unknown) Units (where purchase order units the system is unable to tie to specific commitment orders populate).

  • On Order Units: Confirmed units on order for the period.

    Calculation: (purchase order units not yet received but scheduled for the period)

    When the system policy Assortment Order Type is set to Variant Orders and Demand Management has integrated with an ERP/POM system capable of returning order data with the UUIDs Demand Management generates for each order sent, this field will populate with confirmed purchase order quantities by variant/cluster/period and decrement Committed Units accordingly; in the case where the system policy is set to Inventory Orders and/or external systems are NOT able to return UUIDs by order, purchase order units will appear in the On Order (Unknown) field

  • On Order Exception: The difference between internal commitments and external orders.

    Calculation: Commitment Quantity - ERP On Order Quantity

    Negative numbers indicate that the actual on order quantity is less than the committed quantity.

  • On Order (Unknown): Confirmed units on order for the period which cannot be matched to specific variant/cluster commitment orders

    Calculation: (unlinked purchase order units not yet received but scheduled for the period)

    On Order (Unknown) units will be split across applicable clusters using the same splitting logic as discussed below in EOP Units

    See Committed Units above

  • Total Orders: Sum of all planned and confirmed incoming orders for the period

    Calculation: (Receipt Units + Committed Units + On Order Units + On Order (Unknown) Units)

  • Total Orders AUC: non-editable currency ($1.01); average cost per unit of inbound orders for the period

    Calculation: (Total Orders Cost / Total Orders Units)

  • Total Orders Cost: non-editable currency ($100); cost of inbound orders for the period

    Calculation: (Total Orders Units * Total Orders AUC)

  • Stock Units to Sales Units: non-editable decimal (1.01); ratio expressing the relationship of BOP Units to Sales Units

    Calculation: (BOP Units / Sales Units)

  • Shrink Units: non-editable integer; planned/actual shrink units for the period

    Calculation: From Sales Plan view

  • Inventory Adjustment Units: non-editable integer; planned/actual inventory adjustment units for the period

    Calculation: From Sales Plan view

  • EOP Units: non-editable integer; planned/actual end of period inventory units

    Calculation: (BOP Units – Sales Units + Total Orders – Shrink Units + Inventory Adjustment Units)

    For placeholders/collections for which inventory exists in a DC, the EOP Units of the last actualized period will include a portion of available DC on-hand split out per the following logic:

    Calculation: (Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units) / SUM(Cluster Previous Period EOP Units + Cluster Future Total Orders – Cluster Future Planned Sales Units for all clusters) * DC on-hand units

    While there is an inherent degree of potential inaccuracy in splitting DC on-hand inventory via this method, this was a conscious design decision to assume optimal distribution of available inventory based on future demand by cluster, as the alternative of including complete supply chain logic would have had a significant impact on the performance of the system

  • System Receipt Units: The system recommended receipt units for the period.

    When set to False (default), system policy Allow Receipts After Markdown (Maintain > Policies > Engine: Commitments) prevents the system from ordering additional stock past the planned markdown date. The default is False. Set the system policy to True if additional stock should be ordered beyond the planned markdown date.

    Calculation: Receipt Planning Engine

    this metric seeds System Receipt Units in Sales Plan view as well

  • Ideal EOP Units: non-editable integer; system recommended EOP units for the period

    Calculation: Receipt Planning Engine
