Assortment Plan Defaults tab

Path Code: The code for the corresponding level of the product hierarchy assigned to the current plan.

Path Description: The full description of the product hierarchy path corresponding to the Path Code of the product hierarchy assigned to the current plan.

Cluster Sets: the default cluster sets assigned to new placeholders generated within the specified level of the product hierarchy. Also, sets (along with Additional Allowed Cluster Sets) the cluster sets that will be available in the Cluster Sets parameter when the associated product level is chosen. Available choices are only cluster sets which have been snapshotted and passed validation for use within the current assortment plan

Additional Allowed Cluster Sets: Sets (along with Cluster Sets) the cluster sets that will be available in the Cluster Sets parameter when the associated product level is chosen. Cluster sets selected here are not automatically assigned to new placeholders. Available choices are only cluster sets which have been snapshotted and passed validation for use within the current assortment plan. Cluster sets selected here are not automatically assigned to new placeholders.

If a user attempts to assign multiple cluster sets containing overlapping sites to a single level of the product hierarchy, an error message appears.

Minimum Presentation: Sets the minimum presentation stock in units per site for a given period within the current assortment plan.

This setting will persist across time for any periods for which Minimum Presentation has NOT been set in the Assortment Plan Defaults By Time tab, with values entered there acting as overrides to this value

Seasonality Curve: Sets the default seasonality curve applied to placeholder instances generated within the specified level of the product hierarchy

Only curves of the Manual or Specified Conditions Using Enhanced Filtering types are available for selection within Assortment Planning.

Curve Mode: Sets the mode of the seasonality curve being applied to a placeholder:

  • Static: The selected curve is applied exactly as it exists at the time of application and the placeholder’s curve will not change if changes are made to the curve, whether manually in the Curves module or through dynamic recalculation. Think of this as making a copy of the applied curve
  • Dynamic: Changes made to the curve are pushed down to the placeholder and the dependent sales plans recalculates when those changes are pushed.

Lead Time Days: The number of days prior to a given receipt date that an order must be placed to arrive on time.

Used to calculate Next Order Date in the Main Planning and Receipt Plan views of Line Visualization and Due Date in the Assortment Upcoming Commitments Dashboard for placeholders belonging to the corresponding product hierarchy level for which commitment orders have not been created and which either don’t have variants assigned or their assigned variants lack supplier association.

Number of Markdown Periods: The time frame for the markdown date to phase out date. Determines what the phase out date should be from the markdown date, once a user enters a markdown date during Placeholder creation (i.e. default is 4, markdown date is 4/29, phase out date will be 5/27, four weeks post markdown.). The default is ignored if the markdown falls outside of the phase in/out window or in the past.

  • If Phase Out is set first and a markdown default is set, then Markdown date will auto-set to N default periods before the user's chosen Phase Out.
  • If Phase Out is set first but there is no markdown default for the selected hierarchy level, then markdown date is left empty.
  • If Markdown Date is set first and a markdown default is set, the Phase Out will auto-populate with the N default periods after the user's chosen Markdown Date.
  • If Markdown Date is set first and there is no markdown default, Phase Out will need to be manually entered by the user before saving, or standard phase out logic applies.
  • Any change to either Markdown Date or Phase Out Date after initial value is set does not preserve the default.

Margin Read-only. gross margin percentage for the placeholder. The calculation is: ((AUR ex VAT / (1 + VAT %)) – AUC) / (AUR ex VAT / 1 + VAT %).

ROS $: Read-only. The current ROS $ for the placeholder. Blank for new placeholders.

ROS U: Read-only. The current ROS U for the placeholder. Blank for new placeholders.

Ttl Sls $: Read-only. The current total sales revenue planned for the placeholder. Blank for new placeholders.

Ttl Sls U: Read-only. The current total sales units planned for the placeholder. Blank for new placeholders.

S/T %: Read-only. The target sell-through percentage for the placeholder.

Calculation: (Ttl Sls U / Ttl Buy U)

AUT: user editable currency ($10.01) The home currency AUT (average unit ticket) for the placeholder. If a Map From Variant is specified, then AUT is set to that of the selected variant.

AUT is assumed to be VAT inclusive for the purpose of all calculations within Assortment Planning. For companies in which VAT is not in play for the home currency assigned to the active assortment plan, no special actions are required as long as the associated VAT % is set to zero. For more information, see Currency and VAT Management Overview. For more information, refer to the Currency and VAT Management User Guide.

AUR: Read-only. The AUR (average unit retail price) for the placeholder

Markdown Sales Plan Factor: The ratio between pre-markdown and post-markdown sales. This is used to calculate the post-markdown sales.

Markdown Offset Lock: Links the markdown and phase out dates to each other so that they work dependently on each other. This helps users accurately plan the entire life of the item including markdowns. The standard default mode keeps the markdown and phase out dates independent of each other. Locked mode cannot be enabled on an item with infinite phase out.

The new mode is set at the Placeholder level.

  • If Markdown date changes, Phase Out also changes to keep the specified number of periods between them.
  • If Phase Out changes, then Markdown date also changes to keep the specified number of periods between them.
  • If markdown falls outside of phase in/out window, or in the past, then the default is ignored and the lock-step mode is disabled. Date selectors will take into account the lock step / default range so that the user cannot select themselves into a contradictory state.

Periods of Supply: The periods of supply (period type as defined by the current assortment plan) for placeholders as a minimum stock constraint for receipt engine outputs within the Receipt Plan supporting view of Line Visualization.

Actual by period constraint is defined as the sales planned for this many periods forward from a given period.

This setting interacts with the system policy Assortment Engine DC-to-Store Lead Time (Maintain > Policies > Engine: Commitments).

Assortment Engine DC-to-Store Lead Time is set to 7 days. Periods of Supply is set to 2 with a user planning a receipt to arrive in Wk 1.

For this constraint to be considered fulfilled for Wk 1, the period must end with at least 60 units of inventory on hand to cover weeks 2 and 3. The lead time of 7 days stipulates that stock being received in given week cannot be used to cover the sales planned in that week. Thus, Periods of Supply begins calculating need at Wk 2 rather than Wk 1.

Var Order Multiple: The number by which receipt quantities generated by the Recalculate Receipts functions in the Line Visualization main and receipt plan views will be evenly divisible by. This persists in orders generated from assortment plans within Order Manager. For more information, see:

  • Order Document Details – Break By Size
  • Order Document Details – Prepack

This constraint can also be set in the Placeholder Constraints area of Receipt Plan view.

The system always rounds up to meet Order Multiple constraints. This ensures that the base sales plan on which the receipt plan is built will always be satisfied.

The following is a common scenario where very small parts are sold individually but it is not practical to ship them that way. For a sales plan of 241 units for weeks 1-4, and an Order Multiple of 50 set, the receipt quantity is 250 units or the lowest multiple of 50 that satisfies the entire sales plan.

Size Curve: The size curve assigned to placeholders generated within the specified level of the product hierarchy for the current plan; works in conjunction with Rec Size Min in the Receipt Plan supporting view of Line Visualization; passed to Order Manager for use in the Break by Size stage of the commitment process.

All available size curves populate for selection in this field.

Periods Before Phase In: The default number of periods prior to Phase In date that initial receipt quantities for placeholder instances should be scheduled to be received; used in the Receipt Plan supporting view of Line Visualization. For example, a placeholder with a Phase In date of wk 4 and a Periods Before Phase In of 2 has a default calculated initial receipt for the placeholder of wk 2.

Flow Cadence: The default number of periods between receipt dates calculated by the receipt planning engine; used in the Receipt Plan supporting view of Line Visualization

A placeholder with a Phase In date of wk 3, a Periods Before Phase In of 2, and a Flow Cadence of 4. 3 receipts will occur in wk 1, wk 7, and wk 11.

This is because the purpose of Periods Before Phase In is to ensure product can be in stores by the Phase In date. Thus, the first flow date is considered to be the Phase In date for the purposes of receipt engine calculation.

Min Stk/Sales (Minimum Stock to Sales): The minimum stock to sales ratio for placeholder instances as a minimum stock constraint for receipt engine outputs within the Receipt Plan supporting view of Line Visualization. Actual by period constraint is defined as the sales planned for a given period multiplied by the constraint value (period sales of 10 * Min Stk/Sls of 4 = EOP stock requirement of 40).

Shrink % of Sales: user editable positive percentage to 2 decimal places (1.52%); sets the anticipated shrink units per period as a percentage of planned period sales units for placeholder instances as a minimum stock constraint for receipt engine outputs within the Receipt Plan supporting view of Line Visualization. For example, a period with planned sales of 500 units and a Shrink % of Sales of 2.00% must have an EOP units of at least 10 for this constraint to be satisfied for the period.

Maximum Pack % Total: user editable positive percentage to 2 decimal places (60.00%); defines the maximum % of units in a commitment order of the units within prepacks can represent as a constraint for the prepack optimization engine within Order Manager. When set to less than 100%, units not placed in prepacks will be fulfilled as single units.

I have two placeholder orders for wk 4 for 500 units each. Prepacked together they have a Maximum Pack % Total of 60.00% - the maximum number of units existing in prepacks for this order is 600, however, my prepack output could be less than 600 if the prepack configurations being used don’t evenly roll up to 600 units based on the need calculation by site,

Pack Tolerance: user editable positive percentage to 2 decimal places (5.00%); defines the number of units over the total order quantity the engine may suggest in an attempt to fulfill the Maximum Pack % Total, expressed as a percentage of the total order quantity; used as a constraint for the prepack optimization engine within Order Manager

I have a total order quantity of 1,000 units to be prepacked, a Maximum Pack % Total of 100%, and a Pack Tolerance of 1.00% - the engine may suggest an adjustment of the total order quantity of up to 1010 units to allow for 100% of the original order quantity to be fulfilled by units in prepacks.

when calculating prepack optimization against a Pack Tolerance, the engine will prioritize stores with greater need first

Maximum Packs Per Location: The maximum number of prepacks of any configuration that can be allocated to a single site within the prepack optimization calculation to fulfill the anticipated sales for that site; used as a constraint for the prepack optimization engine within Order Manager

the prepack optimization engine determines that a site requires 50 units from an order being prepacked, Maximum Packs Per Location is set to 5, and the selected prepack configurations all contain 9 units - ignoring site need by size for this example, the engine can only use 5 prepacks, containing 45 units total, as the prepack component of order fulfillment for the site and must assume the remaining 5 units will be fulfilled as single units.

Underage Weight: user editable positive decimal to 2 places (1.25); sets the relative preference for underages when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager

Used in conjunction with Overage Weight and Pack Weighting and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL. Thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).

Overage Weight: user editable positive decimal to 2 places (1.25); sets the relative preference for overages when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager

Used in conjunction with Underage Weight and Pack Weighting and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL. Thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).

Maximum Overage: The number of units per site the prepack optimization engine may over allocate during calculation to reach an acceptable solution; used as a constraint for the prepack optimization engine within Order Manager Maximum Overage %: user editable percentage to 2 decimals (1.25%); sets the percent of total need in units per site the prepack optimization engine may over allocate during calculation to reach an acceptable solution; used as a constraint for the prepack optimization engine within Order Manager

Pack Weighting: user editable positive decimal to 2 places (1.25); sets the relative preference for using the fewest prepack configurations possible when calculating optimal unit allocation by site; used as a constraint for the prepack optimization engine within Order Manager

Used in conjunction with Underage Weight and Overage Weight and calculated as a % to total of the 3 parameters (ie. all 3 are summed then divided by the number of constraints that were not NULL - thus, if only 2 of the 3 are set, the sum of their values is divided by 2 rather than 3 to determine the % to total).

Minimum Units Per Pack: The minimum number of units the prepack optimization engine may suggest in single prepack configurations. Used as a constraint for the prepack optimization engine within Order Manager.

Only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used.

Only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used.

Num Pack Configs: Sets the maximum number of configurations the prepack optimization engine may suggest when determining optimal fulfillment of need within an order. This is used as a constraint for the prepack optimization engine within Order Manager.

Only valid when Optimize Prepack is checked for a given prepack group, otherwise user selected prepack configurations are used

Lower Stock to Sales Threshold: Sets the lower threshold for Stock to Sales. If the STS ratio falls below the lower threshold, then the text is formatted so users can identify it's out of threshold bounds.

Upper Stock to Sales Threshold: Sets the higher threshold for Stock to Sales. If the STS ratio falls above the upper threshold, then the text is formatted so users can identify it's out of threshold bounds.
