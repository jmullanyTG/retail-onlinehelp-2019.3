Setup and Configuration

Before creating assortment plans, the following setup and configuration must be completed:

  • Definition of product and site hierarchies: System Utilities > Views > Maintenance > Hierarchy Filtering.
  • Optional: If you desire to use product and site attributes for the creation of site clusters within the assortment planning process, Attribute Sets, Families, and Values must be defined, and at a minimum, at least one set must be assigned to the topmost level of both the product and site hierarchy: Maintenance > Attribute Management.

    Site and product attribute associations can be set via your configured data import processes and also be managed manually in the following places:

    • Site Attributes: Management > Site Manager > Site Attribute Management
    • Product Attributes: Management > Variant Manager > Variant Attribute Maintenance
  • Optional: If you will be managing the sales plans of ongoing items through the Forecasts module (replenishment forecast integration), it is recommended that you homogenize the dates for the time buckets used in forecasting and the periods to be used in Assortment Planning. It is also recommended that you review the information for Replen Status items to familiarize yourself with the specifics of how such management works between the two modules. Refer to Replen Status.
  • Creation, snapshotting, and validation of cluster sets to used: (Management > Sites > Site Clustering):

    Cluster set validations occur automatically on plan creation and when cluster sets are snapshotted.

  • Definition of Currency Exchange Rates, VAT Rates, and Price Translations:

    DefinitionDescription
    CurrencySystem Utilities > Views > Currency > Currency Exchange Rates
    VATSystem Utilities > Views > Currency > Planning VAT Rate
    Price TranslationsSystem Utilities > Views > Currency > Price Translation Management and Price Translation Hierarchy Assignment

    Users not requiring VAT or Price Translations functionality must still set a rate/factor of 1 (indicating no change) and assign it for all site/product hierarchy levels, as these values are utilized in many system calculations within Assortment Planning.

  • Review/set foundational system policies:

    • More system policies exist than are listed here, but the following are related to foundational data handling within Assortment Planning. Those not listed here are discussed in the specific sections they pertain to. All can be found in Maintenance > Policies > Module: Assortment:

    System PolicyDescription
    Assortment days history to actualizeThe number of days back to go when actualizing sales, receipts and stock on hand for assortment planning. The default is 60.
    Assortment days history to maintain

    The number of days back to maintain detail data for in a live assortment plan. Data older than this date will be summarized and then deleted.  Entering the value "0" will keep all history. The default is 800.

    this policy is useful for database size management, however it should be noted that the amount of historical data maintained has a direct correlation with the data that can be used to seed historical placeholder/variants for in the Seed process. and the date from which to begin seeding cannot exceed the date designated by the current date minus the value set for this policy. For more information on the Seed process, see Seed.

    Assortment Order TypeVariant Orders created from assortment planning are reconciled to individual clusters when coming back as confirmed orders from the host system. Inventory Orders uses a calculated split factor to reconcile inbound order quantities to clusters (see Line Visualization > Supporting Views > Receipt Plan View for details); default is Inventory Orders.
    Days to keep deleted dataThe maximum number of days that deleted data will be held (hidden) in the database.  Set a lower number for more aggressive deleting and more space conservation. The default is 30.
    Default Commitment MergeSets the default merging behavior of order lines when commitment orders are created: 0 = None, 1 = Merge by Variant (see Commitment Orders in Order Manager > General Functionality > Creating Commitment Orders for details); default is 1 (merge by variant).
    Receipt Cluster Merge AllWhen commitments are generated, whether to group receipts for variants together across clusters by default, or whether to keep them separate by default.
